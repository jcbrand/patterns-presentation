Patterns

Design-driven web development through rich, interactive prototypes.

A presentation by JC Brand

“Design isn’t just how it looks, design is how it works.”
Steve Jobs
Why do designers like to work with prototypes?
  • Building to think
  • Learning faster by failing
  • Permission to explore new behaviors

The value of first making a rich, interactive prototype.

  • Communicates the designer's vision to client and rest of team.
  • Initial user-testing and feedback can be gathered.
  • Client knows from the start what they'll be getting, UI and UX-wise.
  • Assumptions can be tested before they're implemented.
  • Design problems are solved during the planning stage.
  • Developers see the context within which their work will be exposed to the end-user.
  • Demonstrations make it easier to sell a website, concept or idea.
So, what is* Patterns? * not a grammar mistake, talking about the toolkit here.

Patterns is a toolkit which allows designers to create rich, interactive prototypes in HTML/CSS without having to learn or write any Javascript

From a designer's perspective

Pick and choose individual patterns, to add functionality and interactivity to your prototype.


More complex behavior can be achieved by combining different patterns.

From a developer's perspective

Patterns lets you write, encapsulate and then expose front-end Javascript features to designers in a way that comes naturally to them.

Individual patterns can do things like:

  • Toggling or switching CSS classes
  • Tabs
  • Ajax page loading
  • Rich text editing
  • Live cloning of page elements
  • Form validation
  • ...pretty much anything that can be done in JS

Create JS modules which configure a DOM parser to check for a certain class name and (optionally) data-* attributes. Such a module will automatically be called when parser finds the markup in the DOM.

There is a pattern generator created with Yeoman, which you can use to generate new patterns. 

yo patternslib pat-example

Advantages of using Patterns

  • Highly reusable.
  • Individual patterns can be reused again and again.
  • Composable
  • Impressive features can be derived by combining patterns in novel ways.
  • Customizable
  • Only use those patterns which you actually need.
  • Allows for graceful degradation.
  • For example, href attribute determines injection source.
  • <!-- yuck -->
<a href="#">Click me!</a>

<!-- nice -->
<a class="pat-inject" href="/path/to/source.html">Click me!</a>
  • Core codebase is small and easy to understand.
  • Scanner, parser and registry code is altogether 620 loc.

When not to use Patterns

  • Single-page apps.
  • Example in Plone is for example the theme editor.
  • Individual patterns as submodules not exposed to the designer.
  • Clunky and misses the point.

Comparison with other JS frameworks or concepts

Fundamental difference is that the intended user of Patterns is the prototype designer.

They shouldn't have to worry about Javascript.

Backbone.js

The Base class from which patterns are usually derived (but don't have to be), is very similar to the Backbone.Model class.


Would also be possible to use Backbone Models and Views inside a pattern, but usually not necessary.


Example of pattern which uses Backbone: pat-converse

Angular.js

Angular has directives.

To a certain degree similar to patterns. Declarative markup which defines widgets and behaviors that should appear in the page.


Can't use them as a designer.

It's not possible to create a functional prototype simply by using and composing Angular directives in your markup without ever having to write any Javascript code.


To do that, you would need to write something like Patterns, in Angular.

React.js

Should be possible to use React views or things like the Virtual DOM and JSX in Patterns.


React and Patterns should be able to live side-by-side in a site.

Webcomponents

Definition:

"A standardised non-obtrusive way to write encapsulated functionality in the form of a component. Each component can hold its own HTML markup, styles and JavaScript code without the risk of breaking things around it."

Similar idea to Patterns.

Declaratively define widgets and enhanced functionality in your HTML.

A Patternslib of webcomponents

Choose the patterns components you want and then download an instantly usable bundle.

Advantages:

  • W3C standard
  • Encapsulation in a "Shadow DOM"
  • The markup for such a widget is encapsulated in its own DOM, with its own CSS rules, which doesn't affect the "normal" DOM around it.

Not ready for prime time.

However, no IE or Safari support (although polyfills exist).

No reason why you can't use both Webcomponents and Patterns in a single project.
