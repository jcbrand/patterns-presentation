Patternslib aims to bring designers and developers together

Design-driven development through the use of prototypes.

Why do designers like to work with prototypes?
“Design isn’t just how it looks, design is how it works.”
Steve Jobs
  • Building to think
  • Learning faster by failing
  • Permission to explore new behaviors

The value of first making a rich, interactive prototype.

  • Communicates the designer's vision to client and rest of team.
  • Assumptions can be tested before they're implemented.
  • Initial user-testing and feedback can be gathered.
  • Design problems are solved during the planning stage.
  • Developers see the context within which their work will be exposed to the end-user.
  • Demonstrations make it easier to sell a website, concept or idea.
So, what is Patternslib?

Patternslib is a toolkit and library which allows designers to create rich, interactive prototypes in HTML/CSS without having to learn or write any Javascript

From a designer's perspective

Pick and choose individual patterns, to add functionality and interactivity to your prototype.


More complex behavior can be achieved by combining different patterns.

From a developer's perspective

Patternslib lets you write, encapsulate and then expose front-end Javascript features to designers in a way that comes naturally to them.

Toggling CSS classes


This is getting toggled

</button>

Switching CSS classes

</ul>

So, as a JS developer, how do I create new patterns?

Let's look at some code.

(function (root, factory) {
    if (typeof define === 'function' && define.amd) {
        define([
            "jquery",
            "pat-base"
        ], function () {
            return factory.apply(this, arguments);
        });
    } else {
        factory ($, Base);
    }
}(this, function ($, Base) {
    'use strict';

    return Base.extend({
        name: 'cycle',
        trigger: ".pat-cycle",

        init: function () {  
            // Your custom code comes here.
        }
    });
}));

Just the custom part

init: function () {  
    var $quotes = this.$el.find('blockquote');
    var $quote = $quotes.eq(Math.floor((Math.random()*$quotes.length)));
    $quote.show();
    setInterval(function () {
        var $newquote = $quote.next('blockquote');
        if (!$newquote.length) {
            $newquote = $quotes.first();
        }
        $quote.fadeOut(function () {
            $newquote.fadeIn();
            $quote = $newquote;
        });
    }.bind(this), 8000);
}

Advantages of using Patterns

  • Highly reusable.
  • Composable
  • Customizable
  • Allows for graceful degradation.
  • For example, href attribute determines injection source.
  • <!-- yuck -->
<a href="#">Click me!</a>

<!-- nice -->
<a class="pat-inject" href="/path/to/source.html">Click me!</a>
  • Core codebase is easy to understand.

When not to use Patterns

  • Single-page apps.
  • Patterns as submodules not exposed to designer.

Comparison with other JS frameworks or concepts

Fundamental difference is that the primary user of Patternslib is the prototype designer.

They shouldn't have to worry about Javascript.

Backbone.js

The Base class from which patterns are usually derived (but don't have to be), is very similar to the Backbone.Model class.


Would also be possible to use Backbone Models and Views inside a pattern, but usually not necessary.


Example of pattern which uses Backbone: pat-converse

Angular.js

Angular has directives.

To a certain degree similar to patterns. Declarative markup which defines widgets and behaviors that should appear in the page.


Can't use them as a designer.

It's not possible to create a functional prototype simply by using and composing Angular directives in your markup without ever having to write any Javascript code.


To do that, you would need to write something like Patterns, in Angular.

React.js

Should be possible to use React views or things like the Virtual DOM and JSX in Patterns.

Webcomponents

Definition:

"A standardised non-obtrusive way to write encapsulated functionality in the form of a component. Each component can hold its own HTML markup, styles and JavaScript code without the risk of breaking things around it."

Similar idea to Patterns.

Not ready for prime time.

Advantages:

  • W3C standard
  • Encapsulation in a "Shadow DOM"

A Patternslib of webcomponents conceivable

Why Patternslib is a good fit for a CMS

  • Progressive enhancement
  • Can be added to older sites
  • Injection works with existing pages and content
  • Non-JS people can add functionality
